# 架构选型 Case 库（通用软件工程）

用于 Stage 3「架构」决策。覆盖通用软件工程 20 大类。AI / Agent 专项见 `architecture-cases-ai.md`；业务自动化场景的落地细节见 `scenario-playbooks.md`。

> **三本 case/playbook 的分工**（v0.4.1 起明确）：
> - 本文件：**通用架构选型**（数字编号 0–20）。回答「在哪一层、用什么形态、付什么代价、退出多痛」。
> - `architecture-cases-ai.md`：**AI / Agent 专项架构选型**（字母编号 A–K）。同样回答架构维度问题，只是 AI 特异。
> - `scenario-playbooks.md`：**业务场景落地**（A–G）。回答「这类业务最小切片是什么、先防什么坑、验证什么算闭环」。
>
> 三者**正交**：架构决策不做场景指南，场景指南不做架构决策。命中重叠话题（例如 LLM 生产链路）时，scenario-playbooks 给切片，architecture-cases-ai 给选型——两边内容不复制，只互相指向。

每个 case 统一格式：

```text
- 一句话定位
- 适用场景
- 不适用场景 / 反例
- 主要代价（运维 / 成本 / 复杂度 / 团队要求）
- 退出成本（迁出去要多痛）
- 常见组合
- 决策信号（出现什么信号该选 / 该换）
- Worked example
- 常见踩坑
```

每大类末尾给一张**决策矩阵表**：行=典型场景，列=候选方案，格=推荐度。

> 推荐度图例：✅ 推荐 / ◯ 可用但有代价 / ⚠️ 谨慎 / ❌ 不推荐

## 0. 形态决策（先于一切）

按 4 步判断项目形态：

1. **用户在哪里运行它？** → Web / 桌面 / 移动 / CLI / 多端
2. **产品真相源在哪里？** → 后端 DB / 本地文件 / 外部平台 / package API
3. **是否需要跨 app 共享代码？** → 决定 monorepo vs polyrepo
4. **是否需要本地 runtime 或进程编排？** → 决定 Desktop+Local Agent vs 纯 Web

| 用户运行环境 | 真相源 | 跨 app 共享 | 本地 runtime | 推荐形态 |
|---|---|---|---|---|
| Web 浏览器 | 后端 DB | 无 | 否 | Web + Backend |
| 桌面 | 本地文件 | 无 | 是 | Desktop + Local Agent |
| 终端/脚本 | 外部 API | 无 | 否 | Python Agent / CLI |
| 多端 | 后端 DB | 是 | 视情况 | Full-stack Monorepo |
| 其他项目 | package API | — | — | Library / SDK |

## 1. Repo 组织

### 1.1 Monorepo（pnpm workspace / Turborepo / Nx / Bazel）

- **一句话定位**：所有相关包/应用住在一个 repo，统一版本与工具链。
- **适用场景**：跨包改动频繁；共享类型/UI/协议；多个 app 同步发版；CI 规模可控。
- **反例**：完全独立的产品线、节奏天差地别、安全/合规要求隔离 repo。
- **主要代价**：CI 时间随仓库膨胀；初期工具链投入（task runner、远程缓存、增量构建）；权限粒度粗。
- **退出成本**：中。拆 repo 时 git 历史可用 `git filter-repo` 拆，但跨包引用需要替换为发布包。
- **常见组合**：pnpm + Turborepo（中小）/ Nx（中大）/ Bazel（极大）；配合 changesets 做版本。
- **决策信号**：
  - 选它：共享代码 ≥ 2 个消费者；前后端契约频繁联动；想 atomic 跨包改动。
  - 换掉：CI > 30 分钟且无法增量；团队彻底独立；外部贡献者只关心一个包。
- **Worked example**：3 人团队做"Web 管理台 + Node BFF + 共享 protocol"。`pnpm-workspace.yaml` 列 `apps/*`、`services/*`、`packages/*`；Turborepo 配 `build/test/lint` pipeline；CI 跑 `turbo run build --filter=...[origin/main]` 只构建受影响包。
- **常见踩坑**：
  - 不配远程缓存导致 CI 慢到无法忍受。
  - `node_modules` 提升导致幽灵依赖；用 pnpm 严格模式。
  - 跨包循环依赖；用 dep-cruiser/`madge` 检测。

### 1.2 Polyrepo（一仓一项目）

- **一句话定位**：每个项目独立 repo，独立 CI、独立版本、独立权限。
- **适用场景**：产品线彼此独立；外部开源组件；强权限隔离；不同合规域。
- **反例**：跨 repo 同步改动频繁；类型契约需要 atomic 修改。
- **主要代价**：跨 repo 协调成本（PR 联动、版本对齐）；契约要靠发版传递；脚手架重复。
- **退出成本**：高（合并 repo 比拆要痛得多）。
- **常见组合**：每仓独立 CI；契约靠发布到内网 npm/PyPI；用 Renovate/Dependabot 自动升级。
- **决策信号**：
  - 选它：各项目用户/团队/发版彼此无关；安全审计要求隔离。
  - 换掉：同一改动需要修 ≥ 3 个 repo 才能闭环。
- **Worked example**：开源 SDK + 商业平台。SDK 单 repo 走 SemVer；平台 repo 通过 npm 引用。
- **常见踩坑**：
  - 契约破坏不打招呼，下游突然炸。强制 SemVer + changelog。
  - 跨 repo 跑 e2e 难；保留一个集成 repo 跑联调。

### 1.3 Hybrid（核心 monorepo + 卫星 polyrepo）

- **一句话定位**：核心产品在 monorepo；外部 SDK、独立产物、合规隔离的部分单独成 repo。
- **适用场景**：核心团队需要 atomic 联动，但要对外发可独立消费的包/服务。
- **反例**：团队体量小，hybrid 反而双倍工具链负担。
- **主要代价**：边界判断成本；卫星 repo 与核心契约同步要靠发版。
- **退出成本**：低（哪边偏向哪边都能逐步迁）。
- **常见组合**：核心用 Turborepo + changesets 发包；卫星 repo 消费已发布版本。
- **决策信号**：开源/对外的 SDK 必须单仓发；合规域需要隔离。
- **Worked example**：核心 monorepo 含 web/backend/protocol；CLI SDK 卫星 repo 引用 protocol 的发布版本。
- **常见踩坑**：卫星仓忘记升级核心 protocol，长期落后。设置 staleness 警告。

#### 决策矩阵 — Repo 组织

| 场景 | Monorepo | Polyrepo | Hybrid |
|---|---|---|---|
| 单团队 Web + BFF + 共享类型 | ✅ | ⚠️ | ◯ |
| 多产品线、各自团队 | ⚠️ | ✅ | ◯ |
| 开源 SDK + 商业平台 | ⚠️ | ✅ | ✅ |
| 跨包重构频繁 | ✅ | ❌ | ◯ |
| 合规要求隔离 | ❌ | ✅ | ✅ |
| 团队 < 5 人 | ✅ | ◯ | ⚠️ |

## 2. 前端渲染模式

### 2.1 CSR（SPA）

- **一句话定位**：浏览器拿到空 HTML 后由 JS 渲染。
- **适用场景**：内部管理台、登录后高交互应用、SEO 不重要。
- **反例**：公开内容站、营销页、SEO 关键。
- **主要代价**：首屏白屏；SEO 困难；大 bundle 影响弱网。
- **退出成本**：中。要改 SSR/SSG 通常意味着切框架或大改路由层。
- **常见组合**：Vite + React Router；TanStack Query 管远程数据。
- **决策信号**：
  - 选它：登录墙之后；高度交互；SEO 无所谓。
  - 换掉：营销/内容页；SEO/分享卡片需求。
- **Worked example**：B 端管理台。Vite + React + TanStack Query；首屏 loading skeleton + 路由懒加载。
- **常见踩坑**：把营销首页也塞进 SPA，SEO/首屏全坏；该首页应单独 SSR/SSG。

### 2.2 SSR（服务端渲染）

- **一句话定位**：每次请求服务端渲染 HTML。
- **适用场景**：内容动态、SEO 重要、强个性化、首屏关键。
- **反例**：纯静态内容；流量极大且内容相同（应 SSG/CDN）。
- **主要代价**：服务端运算成本；缓存策略复杂；冷启动（serverless 上）。
- **退出成本**：中（迁回 CSR/SSG 可行，但 SEO 与首屏会回退）。
- **常见组合**：Next.js / Nuxt / Remix / SvelteKit；前置 CDN 做缓存。
- **决策信号**：
  - 选它：电商、媒体、SaaS marketing；SEO/首屏直接影响转化。
  - 换掉：内容彻底静态；流量稳定可全量 SSG。
- **Worked example**：电商商品详情。Next.js App Router，详情页 SSR + ISR 60s revalidate；CDN 命中率 > 90%。
- **常见踩坑**：把鉴权后的高个性化页面 SSR，缓存命中率几乎 0，反而比 CSR 慢。

### 2.3 SSG（静态站点生成）

- **一句话定位**：构建期生成 HTML，部署到 CDN。
- **适用场景**：博客、文档、营销页、内容更新低频。
- **反例**：高度个性化；高频内容更新（每分钟级）。
- **主要代价**：内容更新需要重新构建（除非搭 ISR）；构建时长随页面数线性增长。
- **退出成本**：低。
- **常见组合**：Astro / Next SSG / Hugo / VitePress；CDN 直接托管。
- **决策信号**：
  - 选它：内容主导、SEO 重要、个性化少。
  - 换掉：内容频繁更新且页面 ≥ 万级；构建时长爆炸。
- **Worked example**：开源项目文档站。VitePress + GitHub Pages；每次 push 触发构建。
- **常见踩坑**：把搜索/个性化做成全静态，结果靠 client-only 重新发请求，反而退化为 CSR。

### 2.4 ISR（增量静态再生）

- **一句话定位**：SSG 的页面允许在背景按周期或按需重新生成。
- **适用场景**：电商、新闻、内容大量但更新频率适中。
- **反例**：实时数据（应 SSR）；完全不更新（纯 SSG 即可）。
- **主要代价**：缓存键设计与 invalidation 复杂；调试困难。
- **退出成本**：低（可退到 SSG 或 SSR）。
- **常见组合**：Next.js（Vercel 原生）；自建需要 KV 缓存层。
- **决策信号**：页面数大、更新有但非实时、想要 SSG 的速度 + SSR 的新鲜度。
- **Worked example**：电商商品页 `revalidate: 300`，下单后 webhook 触发 on-demand revalidate。
- **常见踩坑**：自建 ISR 缓存一致性翻车；尽量用平台原生。

### 2.5 RSC / Islands（部分服务端组件 / 岛屿）

- **一句话定位**：默认服务端渲染，标注的"岛"才水合为客户端组件。
- **适用场景**：内容为主、少量交互；想要小 JS 包 + 服务端数据访问。
- **反例**：全应用高交互（SPA 仍更直接）；团队不熟悉 server/client 边界。
- **主要代价**：心智模型新；server-only / client-only 边界违规会运行时报错；生态尚在演进。
- **退出成本**：中。
- **常见组合**：Next.js App Router（RSC）/ Astro（Islands）。
- **决策信号**：
  - 选它：内容站点带局部交互；想要默认零 JS。
  - 换掉：全交互后台；团队熟悉度不足时不要冒进。
- **Worked example**：媒体站，文章默认 RSC，仅评论组件标 `'use client'`。
- **常见踩坑**：在 RSC 里 `useState`、把 secret 渲染到 client。

#### 决策矩阵 — 渲染模式

| 场景 | CSR | SSR | SSG | ISR | RSC/Islands |
|---|---|---|---|---|---|
| 内部管理台 | ✅ | ◯ | ❌ | ❌ | ⚠️ |
| 公开 SaaS 营销 + 应用 | ◯ | ✅ | ◯ | ✅ | ✅ |
| 文档/博客 | ❌ | ◯ | ✅ | ◯ | ✅ |
| 电商详情 | ⚠️ | ✅ | ◯ | ✅ | ✅ |
| 高度实时交易 | ⚠️ | ✅ | ❌ | ❌ | ⚠️ |

## 3. 前后端关系

### 3.1 前后端分离（独立部署 + REST/GraphQL）

- **一句话定位**：前端纯 SPA/App；后端独立服务对外暴露 API。
- **适用场景**：客户端有多端（Web/Mobile/Desktop）；前后端团队边界清晰。
- **反例**：单 Web 客户端 + 强 SEO；初期小团队（额外协调成本）。
- **主要代价**：契约维护、CORS、鉴权 token 流转、跨域 cookie 难度。
- **退出成本**：低-中。
- **常见组合**：React/Vue + FastAPI/NestJS/Spring；OpenAPI 生成 client。
- **决策信号**：
  - 选它：≥ 2 个客户端复用同一 API。
  - 换掉：仅一个 Web 客户端且 SEO 重要 → 改全栈框架。
- **Worked example**：Web 后台 + Mobile App 共用 FastAPI；用 OpenAPI codegen 出 TS/Swift client。
- **常见踩坑**：契约靠口口相传，前后端类型漂移。强制 OpenAPI/Protobuf。

### 3.2 BFF（Backend for Frontend）

- **一句话定位**：每种前端有专属薄后端，聚合下游服务，按客户端形态裁剪数据。
- **适用场景**：多端差异大；下游有多个微服务/外部 API；想减少客户端 N+1 请求。
- **反例**：单端 + 单后端，BFF 是冗余层。
- **主要代价**：多一层；BFF 自身的运维与版本管理。
- **退出成本**：中。
- **常见组合**：Node BFF + GraphQL / tRPC；下游 REST/gRPC。
- **决策信号**：
  - 选它：客户端聚合调用 ≥ 3 个下游；移动端流量金贵需裁剪。
  - 换掉：下游收敛为单服务后；BFF 沦为透传。
- **Worked example**：Web BFF（GraphQL）+ Mobile BFF（REST 精简）共享下游订单/用户/商品微服务。
- **常见踩坑**：BFF 写死业务逻辑，演变成另一个单体。

### 3.3 全栈框架内聚（Next.js / Nuxt / Remix / SvelteKit）

- **一句话定位**：前后端在一个框架里，路由即 API。
- **适用场景**：单 Web 客户端；SEO/首屏重要；小团队全栈。
- **反例**：多端（Mobile/Desktop）需要复用 API；后端逻辑复杂超出 framework 能力。
- **主要代价**：和框架耦合；后端能力受框架限制；扩到多端时要剥离 API。
- **退出成本**：中-高（API 层和 framework 绑定深）。
- **常见组合**：Next.js + Prisma + Vercel；Remix + 自托管 Node。
- **决策信号**：
  - 选它：MVP/创业初期；只有 Web；SEO 重要。
  - 换掉：要做 Mobile App；后端 worker/复杂任务越来越多。
- **Worked example**：SaaS MVP 用 Next.js App Router，路由内 server actions 直连 DB；Stripe webhook 走 `/api/webhooks/stripe`。
- **常见踩坑**：把重计算/长任务放进 serverless function，超时 + 冷启动炸。

### 3.4 API Gateway 聚合

- **一句话定位**：网关统一入口，做路由、鉴权、限流，下游是多个独立服务。
- **适用场景**：微服务规模化；多客户端共用同一边界；统一鉴权/限流/审计。
- **反例**：服务数量少；网关引入过早导致瓶颈和单点。
- **主要代价**：网关本身的可用性、配置复杂度、调试链路变长。
- **退出成本**：中。
- **常见组合**：Kong / APISIX / Envoy / AWS API Gateway。
- **决策信号**：服务数 ≥ 5；统一限流/鉴权诉求强。
- **Worked example**：Kong 前置鉴权 + 限流，下游订单/用户/搜索服务用 mTLS 互通。
- **常见踩坑**：把业务逻辑写进网关插件，难维护、难灰度。

#### 决策矩阵 — 前后端关系

| 场景 | 前后端分离 | BFF | 全栈框架 | API Gateway |
|---|---|---|---|---|
| Web-only MVP | ◯ | ❌ | ✅ | ❌ |
| Web + Mobile | ✅ | ✅ | ⚠️ | ◯ |
| 微服务规模化 | ✅ | ✅ | ❌ | ✅ |
| 内容站 + 营销 | ◯ | ❌ | ✅ | ❌ |
| 多客户端 + 多下游 | ◯ | ✅ | ❌ | ✅ |

## 4. 后端架构

### 4.1 单体（Monolith）

- **一句话定位**：一个进程、一份代码、一个数据库。
- **适用场景**：MVP；团队 < 10；领域不清晰；强一致性要求。
- **反例**：多团队并行开发同一单体；不同模块流量/扩展模式截然不同。
- **主要代价**：代码膨胀后修改易耦合；扩展只能整体扩。
- **退出成本**：拆微服务很痛（数据库、跨模块调用、事务）。
- **常见组合**：FastAPI/NestJS/Rails/Django + Postgres + Redis。
- **决策信号**：
  - 选它：领域未稳；团队 < 10；快速迭代优先。
  - 换掉：多团队冲突；模块流量/SLA 差异巨大。
- **Worked example**：早期 SaaS 用 FastAPI 单体，分目录 `modules/<domain>/`，DB 单库分 schema。
- **常见踩坑**：模块边界没建好，后期想拆拆不动。

### 4.2 模块化单体（Modular Monolith）

- **一句话定位**：物理上单进程，逻辑上严格模块边界，模块只通过定义良好的接口交互。
- **适用场景**：希望保留单体运维简单，又为未来拆服务留好缝。
- **反例**：团队不愿守边界；老代码缠绕严重难以梳理。
- **主要代价**：纪律成本（团队必须守模块边界）；工具链支持有限。
- **退出成本**：低（边界已清晰，拆服务就是把模块抽出来）。
- **常见组合**：Domain modules + dependency rules（ArchUnit / 自定义 lint）。
- **决策信号**：领域逐渐清晰但还不到拆服务；团队要求未来灵活性。
- **Worked example**：Spring Boot 项目按 `order/`、`user/`、`payment/` 模块切，依赖规则用 ArchUnit 强制。
- **常见踩坑**：边界靠人守，review 不严立即破防。配自动化检查。

### 4.3 微服务

- **一句话定位**：每个领域独立服务，独立部署、独立数据。
- **适用场景**：多团队并行；不同模块伸缩/SLA 差异大；有运维平台支撑。
- **反例**：团队 < 20；DevOps 能力不足；强一致性事务需求多。
- **主要代价**：分布式事务、跨服务调用链、运维平台、可观测性投入。
- **退出成本**：高（合并回单体很痛）。
- **常见组合**：K8s + Service Mesh（Istio/Linkerd）+ 集中可观测性 + Saga/Outbox。
- **决策信号**：
  - 选它：多团队；多 SLA；多语言诉求。
  - 换掉：服务数膨胀但实际只是被切碎的单体；调用链深难调试。
- **Worked example**：电商 4 团队拆 4 服务（订单/库存/支付/搜索）+ Kafka 事件总线 + OpenTelemetry。
- **常见踩坑**：
  - 把单体按表拆成服务，数据库还是共享 → "分布式单体"。
  - 同步调用链 4 层以上，下游一抖全炸。

### 4.4 Serverless / FaaS

- **一句话定位**：按请求/事件触发的无状态函数，平台管运维与扩缩容。
- **适用场景**：流量极不均匀；事件驱动；初创快上线；后台批处理。
- **反例**：长任务（> 平台超时）；高频调用（成本反超 VM）；强本地状态；冷启动敏感。
- **主要代价**：冷启动；vendor lock-in；调试与本地开发不直观；超时限制。
- **退出成本**：高（平台 SDK 与触发器迁移痛）。
- **常见组合**：AWS Lambda + API Gateway + DynamoDB；Cloudflare Workers + KV/D1；Vercel Functions。
- **决策信号**：
  - 选它：事件触发；流量稀疏；团队不想运维。
  - 换掉：函数链路 > 5 步；冷启动影响 SLO；月成本超 VM 等价。
- **Worked example**：图像处理：S3 上传触发 Lambda 生成缩略图回写 S3。
- **常见踩坑**：把 Web 应用全塞 Lambda，冷启动让 SaaS 体验崩溃；该用容器。

### 4.5 事件驱动 / CQRS

- **一句话定位**：写入走命令并发事件，读侧由事件投影出物化视图。
- **适用场景**：读写比悬殊；审计/可追溯强需求；多消费者订阅同一事实。
- **反例**：CRUD 简单系统；团队不熟悉异步思维。
- **主要代价**：最终一致性心智；事件 schema 演化；调试链路复杂。
- **退出成本**：高（数据模型与查询彻底重塑）。
- **常见组合**：Kafka + Outbox + CDC（Debezium）；读侧 Postgres/ES。
- **决策信号**：
  - 选它：金融/审计；多读侧消费；写少读多。
  - 换掉：事件量小且无审计需求；团队 onboarding 困难。
- **Worked example**：支付系统记录"PaymentAuthorized/Captured/Refunded"事件，账务/风控/通知各自消费。
- **常见踩坑**：事件 schema 不版本化，下游消费者炸；用 schema registry。

#### 决策矩阵 — 后端架构

| 场景 | 单体 | 模块化单体 | 微服务 | Serverless | 事件驱动 |
|---|---|---|---|---|---|
| MVP / 团队 < 10 | ✅ | ✅ | ❌ | ◯ | ❌ |
| 领域清晰 + 多团队 | ⚠️ | ✅ | ✅ | ◯ | ◯ |
| 流量极不均匀 | ⚠️ | ◯ | ◯ | ✅ | ✅ |
| 强审计/可追溯 | ◯ | ◯ | ◯ | ◯ | ✅ |
| 简单 CRUD | ✅ | ✅ | ❌ | ◯ | ❌ |

## 5. 数据架构

### 5.1 单库单实例（Postgres/MySQL）

- **一句话定位**：一个 RDBMS 实例承载全部业务数据。
- **适用场景**：MVP、读写量适中、强一致性事务多。
- **反例**：多租户隔离要求强；QPS/数据量超出单机。
- **主要代价**：单点故障；扩展受限。
- **退出成本**：低-中（迁到主从/分库都有路径）。
- **常见组合**：Postgres + 备份 + read replica。
- **决策信号**：
  - 选它：MVP；数据量 < 数百 GB；QPS < 数千。
  - 换掉：写入接近瓶颈；数据量爆炸。
- **Worked example**：早期 SaaS 用 RDS Postgres + 自动备份 + 1 个 read replica 灾备。
- **常见踩坑**：从未压测过单库瓶颈，等出事才慌；定期容量回顾。

### 5.2 读写分离 / 主从

- **一句话定位**：写主、读从；读靠从库分担。
- **适用场景**：读多写少；报表/列表查询拖慢主库。
- **反例**：写密集；强读后写一致性需求。
- **主要代价**：复制延迟带来一致性问题；ORM 路由。
- **退出成本**：低。
- **常见组合**：Postgres logical/physical replication；HAProxy/PgBouncer。
- **决策信号**：读 QPS 远高于写；报表查询影响主库。
- **Worked example**：列表/报表走 read replica；下单走主库。
- **常见踩坑**：用户下单后立刻读列表，落到从库读不到；用 read-your-writes 标记或路由。

### 5.3 分库分表 / Sharding

- **一句话定位**：按 key 切分数据到多个库/表。
- **适用场景**：单表行数十亿；写入超出单机；多租户大客户隔离。
- **反例**：数据量未到瓶颈；跨 shard 查询多。
- **主要代价**：跨 shard 事务/JOIN 复杂；rebalancing 痛苦；运维大幅复杂。
- **退出成本**：极高。
- **常见组合**：Vitess / Citus / 应用层 sharding；按 tenant_id / user_id 切。
- **决策信号**：
  - 选它：单表 > 10 亿行；写 QPS 单机扛不住。
  - 换掉：上之前先穷尽缓存/读写分离/归档。
- **Worked example**：消息系统按 conversation_id sharding 到 32 个 Postgres。
- **常见踩坑**：sharding key 选错，热点全压一个 shard。

### 5.4 多租户（共享库 / 独立 schema / 独立库）

- **一句话定位**：B 端 SaaS 的租户数据隔离方式。
- **三档对比**：
  - 共享库共享表（tenant_id 列）：成本最低、隔离最弱。
  - 共享库独立 schema：中等成本、中等隔离、Postgres 友好。
  - 独立库：最强隔离、最高成本。
- **决策信号**：
  - 共享：海量小租户。
  - 独立 schema：中型 B 端，合规要求中等。
  - 独立库：大客户、强合规、定制需求。
- **常见踩坑**：起步用共享列，后期想升级独立 schema 时数据迁移痛；大客户预先支持"专属库"。

### 5.5 关系型 + 缓存（Redis）

- **一句话定位**：关键热数据放 Redis，DB 兜底。
- **适用场景**：读热点、会话、计数、排行榜。
- **反例**：把 Redis 当主数据库；强一致性写。
- **主要代价**：缓存一致性（更新策略 / 失效）；多一层运维。
- **退出成本**：低。
- **常见组合**：Cache-aside / Write-through / Read-through；TTL + 主动失效。
- **常见踩坑**：缓存击穿/穿透/雪崩；用单飞、空值缓存、随机 TTL。

### 5.6 关系型 + 搜索（Elasticsearch / Meilisearch / OpenSearch）

- **一句话定位**：全文搜索/聚合走专用引擎，DB 是真相源。
- **适用场景**：全文搜索、faceted 过滤、复杂聚合。
- **反例**：把搜索引擎当主存（无事务、最终一致）。
- **主要代价**：双写一致性；schema 映射；运维。
- **退出成本**：中。
- **常见组合**：DB → CDC（Debezium）→ ES；或写双写。
- **常见踩坑**：双写不一致漂移；用 outbox / CDC 而不是应用层双写。

### 5.7 文档型 / KV / 时序 / 图

- **MongoDB / DynamoDB**：schema 灵活、嵌套数据；不擅长跨集合事务。
- **Redis / DynamoDB / KV**：极简访问；适合会话、配置、计数。
- **InfluxDB / TimescaleDB**：时序数据；指标/IoT。
- **Neo4j / 图库**：强关系查询；社交、知识图谱、风控网络。
- **决策信号**：先问"我的查询是什么形状"，再选库。形状不匹配，性能/成本都崩。
- **常见踩坑**：用 Mongo 当 RDBMS 做强事务；用 Postgres 跑大规模时序。

### 5.8 数据湖 / 仓（Snowflake / BigQuery / ClickHouse / Iceberg）

- **一句话定位**：分析与报表的真相源，OLTP 数据 ETL/CDC 进来。
- **适用场景**：BI、用户行为分析、机器学习特征。
- **反例**：当线上业务库用。
- **主要代价**：ETL 链路；成本（按扫描/存储计）。
- **决策信号**：报表查询拖垮 OLTP；分析需求增多。
- **常见踩坑**：把 PII 直接灌进数仓没脱敏；治理体系要先行。

#### 决策矩阵 — 数据架构（按场景推荐主存）

| 场景 | 单库 | 主从 | 分库分表 | Document | KV | 搜索引擎 | 时序 | 图 | 数仓 |
|---|---|---|---|---|---|---|---|---|---|
| MVP CRUD | ✅ | ◯ | ❌ | ◯ | ❌ | ❌ | ❌ | ❌ | ❌ |
| 读多写少 + 报表 | ◯ | ✅ | ❌ | ❌ | ❌ | ◯ | ❌ | ❌ | ✅(分析) |
| 海量写 | ❌ | ❌ | ✅ | ◯ | ◯ | ❌ | ◯ | ❌ | ❌ |
| 全文搜索 | ❌ | ❌ | ❌ | ❌ | ❌ | ✅ | ❌ | ❌ | ❌ |
| IoT/监控 | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ | ✅ | ❌ | ◯ |
| 关系网络 | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ | ✅ | ❌ |

## 6. 通信与协议

### 6.1 REST

- **一句话定位**：以资源为中心、HTTP 动词驱动的接口风格。
- **适用场景**：开放 API、前后端通用、缓存友好（GET）。
- **反例**：实时双向；强类型契约严格场景。
- **主要代价**：N+1 查询常见；客户端聚合需要多个请求。
- **常见组合**：OpenAPI 描述 + codegen；HTTP 缓存。
- **常见踩坑**：把 RPC 风格塞 REST，端点变成动词；用 GraphQL/RPC 更合适。

### 6.2 GraphQL

- **一句话定位**：客户端按需查询、单端点。
- **适用场景**：多端聚合、客户端字段差异大。
- **反例**：简单内部 API；缓存策略复杂；团队不熟。
- **主要代价**：N+1（dataloader 必备）；权限/限流复杂；schema 治理。
- **决策信号**：多端 + 多组合查询；客户端不可控（外部开发者）。
- **常见踩坑**：暴露任意字段未做 cost limit，被 query 风暴搞挂。

### 6.3 gRPC

- **一句话定位**：基于 HTTP/2 + Protobuf 的强类型 RPC。
- **适用场景**：服务间内部通信；多语言；低延迟。
- **反例**：浏览器直接调用（需 grpc-web）；调试要专用工具。
- **主要代价**：浏览器支持差；二进制不便排查。
- **常见组合**：Service Mesh + gRPC + buf 生成。
- **常见踩坑**：用 gRPC 做对外 API，外部接入痛。

### 6.4 tRPC

- **一句话定位**：TypeScript 端到端类型安全 RPC。
- **适用场景**：纯 TS 单仓；前后端一锅端。
- **反例**：多语言；要求语言无关契约；公开 API。
- **主要代价**：与 TS 强绑定；不适合外部消费者。
- **常见踩坑**：为了 tRPC 把后端锁死在 Node。

### 6.5 WebSocket / SSE

- **一句话定位**：长连接推送；WebSocket 双向，SSE 单向（服务端→客户端）。
- **适用场景**：实时通知、协作编辑、行情、AI 流式输出。
- **反例**：偶尔推送（用 polling 即可）；CDN/缓存不友好。
- **主要代价**：连接管理、横向扩展（sticky/分布式 pub-sub）；网络中断重连。
- **常见组合**：SSE for AI streaming；WebSocket for collab；Redis Pub/Sub 或 NATS 做扇出。
- **常见踩坑**：SSE 不走 HTTP/2 时浏览器并发连接限制。

### 6.6 消息队列（Kafka / RabbitMQ / Redis Streams / NATS）

- **Kafka**：高吞吐、持久化日志、事件溯源。
- **RabbitMQ**：复杂路由、低延迟任务队列。
- **Redis Streams**：轻量、与 Redis 一体。
- **NATS / JetStream**：轻量+持久化可选、云原生。
- **决策信号**：吞吐 → Kafka；任务路由 → RabbitMQ；轻量 → Redis；云原生轻量持久 → NATS。
- **常见踩坑**：Kafka 当 RPC 用；RabbitMQ 当事件溯源用。

#### 决策矩阵 — 通信协议

| 场景 | REST | GraphQL | gRPC | tRPC | WS/SSE | MQ |
|---|---|---|---|---|---|---|
| 公开 API | ✅ | ✅ | ⚠️ | ❌ | ◯ | ❌ |
| 服务间内网 | ◯ | ◯ | ✅ | ❌ | ❌ | ✅ |
| 单仓 TS 全栈 | ◯ | ◯ | ❌ | ✅ | ◯ | ◯ |
| 实时流式 | ❌ | ◯(sub) | ◯ | ❌ | ✅ | ✅(扇出) |
| 多端聚合 | ◯ | ✅ | ❌ | ◯ | ❌ | ❌ |

## 7. 鉴权与身份

### 7.1 Session + Cookie

- **一句话定位**：服务端会话存储 + HttpOnly cookie。
- **适用场景**：同源 Web；强 CSRF 防护；后台。
- **反例**：跨域多端；移动 App。
- **主要代价**：会话存储；横向扩展需共享存储。
- **常见踩坑**：未设 SameSite/Secure/HttpOnly。

### 7.2 JWT（Access + Refresh）

- **一句话定位**：无状态 token，access 短、refresh 长。
- **适用场景**：跨域、移动端、微服务间传递身份。
- **反例**：要求即时撤销（JWT 撤销难）。
- **主要代价**：撤销/黑名单复杂；token 体积；密钥轮换。
- **常见踩坑**：access token 过长（> 1h）；refresh 不旋转。

### 7.3 OAuth2 / OIDC

- **一句话定位**：标准化的第三方授权 / 单点登录。
- **适用场景**：企业 SSO；社交登录；开放平台。
- **反例**：内部小系统单一身份源（用 session 即可）。
- **主要代价**：标准复杂；正确实现成本高。
- **常见踩坑**：自己实现 OAuth provider；优先用托管。

### 7.4 第三方托管（Auth0 / Clerk / Supabase Auth / Cognito）

- **一句话定位**：把鉴权交给托管服务。
- **适用场景**：初创、想要 SSO/MFA/合规快速上。
- **反例**：高度定制流程；预算敏感+用户量极大；强数据主权。
- **主要代价**：vendor lock-in；按 MAU 计费；定制能力受限。
- **退出成本**：中-高（用户数据迁移合规要求复杂）。
- **常见踩坑**：把业务用户表完全托管在第三方，自家 DB 没有 user 主表 → 后续做用户分析痛。保留本地 user 表 + 外部 idp_id 映射。

### 7.5 自建 IdP（Keycloak / Ory / Authentik）

- **一句话定位**：自部署的开源身份平台。
- **适用场景**：大体量、强定制、合规要求自托管。
- **反例**：小团队、无运维。
- **主要代价**：运维与升级；高可用部署。
- **常见踩坑**：Keycloak 升级跨大版本数据库迁移踩坑。

### 7.6 服务间（API Key / mTLS / SPIFFE）

- **API Key**：简单内部脚本/低敏。
- **mTLS**：服务网格内强身份。
- **SPIFFE/SPIRE**：跨集群身份联邦。
- **常见踩坑**：API Key 永久有效不轮换。

#### 决策矩阵 — 鉴权

| 场景 | Session | JWT | OAuth2 | 托管 IdP | 自建 IdP |
|---|---|---|---|---|---|
| 同源 Web 后台 | ✅ | ◯ | ❌ | ◯ | ◯ |
| 跨域 / Mobile | ◯ | ✅ | ◯ | ✅ | ◯ |
| 企业 SSO | ◯ | ◯ | ✅ | ✅ | ✅ |
| 创业快上线 | ◯ | ◯ | ❌ | ✅ | ❌ |
| 强数据主权 | ✅ | ✅ | ◯ | ❌ | ✅ |

## 8. 部署与运行环境

### 8.1 容器（Docker + K8s / ECS / Nomad）

- **一句话定位**：容器化部署 + 编排器调度。
- **适用场景**：多服务；需弹性；团队有 DevOps。
- **反例**：单应用 + 小团队（PaaS 更划算）。
- **主要代价**：K8s 学习曲线；运维平台投入。
- **退出成本**：中（容器化产物可移植，但 manifests 平台相关）。
- **常见组合**：K8s + Helm/Kustomize + Argo CD + Prom/Grafana。
- **常见踩坑**：上 K8s 但服务只有 2 个；ROI 极低，先 ECS/Fargate/Render。

### 8.2 PaaS（Vercel / Netlify / Render / Fly.io / Railway）

- **一句话定位**：托管平台帮你管运行时与扩缩容。
- **适用场景**：快速上线；前端 + 轻后端；小团队。
- **反例**：强合规；超大流量成本反超；底层定制。
- **主要代价**：vendor lock-in；调试受限；价格非线性。
- **退出成本**：低-中（前端易迁，含平台特有 SDK 时偏中）。
- **常见踩坑**：所有逻辑塞 serverless function；超时与冷启动事故。

### 8.3 传统 VM

- **一句话定位**：直接在虚机上跑进程或 systemd。
- **适用场景**：少量长生命周期服务；强稳定；遗留系统。
- **反例**：弹性需求强；多服务编排。
- **主要代价**：手工运维；扩展不便。
- **常见踩坑**：用 cron 做关键任务无监控；任务跑挂没人知。

### 8.4 Serverless

- **一句话定位**：按请求计费的无服务器函数。
- **见 4.4**。

### 8.5 桌面分发（Tauri / Electron + 签名公证）

- **一句话定位**：把 Web 技术打成桌面应用，附带本地能力。
- **适用场景**：需要本地文件/进程/离线；分发到员工或终端用户桌面。
- **反例**：纯展示用 Web 即可。
- **主要代价**：签名/公证；自动更新；多平台测试。
- **退出成本**：中（壳层与本地能力分离则可拆）。
- **常见组合**：Tauri（小包+Rust 后端）/ Electron（生态成熟）；Sentry crash report；Sparkle/自家更新通道。
- **决策信号**：
  - Tauri：包大小敏感、安全要求高、Rust 能力可用。
  - Electron：生态依赖重、迭代速度优先。
- **常见踩坑**：忘做签名 → macOS Gatekeeper 拒绝；用户首次打开极差体验。

### 8.6 移动分发（App Store / TestFlight / Play）

- **一句话定位**：通过商店分发 native/跨端 App。
- **主要代价**：审核周期；强制 hotfix 困难（用 OTA bundle 受限）。
- **常见踩坑**：苹果合规规则更新打回；预留 buffer。

| 多服务规模化 | ✅ | ◯ | ⚠️ | ◯ | ❌ | ❌ |

## 9. 边缘与 CDN

### 9.1 纯 CDN

- **适用**：静态资源、图片、视频；HTML 命中率高。
- **常见组合**：Cloudflare / Fastly / CloudFront。

### 9.2 边缘计算（Cloudflare Workers / Vercel Edge / Lambda@Edge）

- **适用**：地理就近的轻处理：A/B、个性化、鉴权、重写、地理分发。
- **反例**：重计算、连关系数据库、长连接。
- **主要代价**：运行时受限（无完整 Node 生态）；冷启动很短但仍有；连 DB 通常需要边缘 DB（D1/Turso/Neon serverless driver）。
- **常见踩坑**：把整套业务塞边缘，DB 网络往返反而更慢。

### 9.3 区域多活

- **适用**：跨大洲业务；强 SLA。
- **代价**：多区数据一致性、复杂度极高。
- **常见踩坑**：写冲突没设计好。先单区主写、跨区只读起步。

#### 决策矩阵 — 边缘

| 场景 | 纯 CDN | 边缘函数 | 多活 |
|---|---|---|---|
| 静态资产/媒体 | ✅ | ◯ | ◯ |
| 个性化/AB | ◯ | ✅ | ◯ |
| 全球低延迟读 | ◯ | ✅ | ✅ |
| 全球低延迟写 | ❌ | ⚠️ | ✅(代价高) |

## 10. 异步与后台任务

### 10.1 进程内队列

- **适用**：单实例小任务；不持久。
- **反例**：进程重启会丢；多实例不一致。

### 10.2 Redis-based（BullMQ / RQ / Sidekiq）

- **适用**：中等可靠性、易用；任务量中等。
- **代价**：依赖 Redis 持久化；故障恢复策略要明确。
- **常见踩坑**：未配置 visibility timeout，重复消费。

### 10.3 专用 Broker（Kafka / RabbitMQ / SQS）

- **适用**：高吞吐、可靠投递、跨服务事件。
- **代价**：运维或服务费。

### 10.4 工作流引擎（Temporal / Airflow / Prefect / Step Functions）

- **Temporal**：长周期、可补偿的业务流程（订单、退款、入职流程）。
- **Airflow / Prefect / Dagster**：批处理 ETL，DAG 调度。
- **Step Functions**：AWS 原生 + 短流程。
- **决策信号**：
  - 业务流程长 + 需补偿 + 可重放 → Temporal。
  - 数据 ETL 调度 → Airflow/Prefect/Dagster。
- **常见踩坑**：用普通队列做长事务，状态管理失控；用工作流引擎。

### 10.5 Cron / Scheduled

- **适用**：简单周期任务。
- **代价**：观测/失败重试要自己加。
- **常见踩坑**：单机 cron 单点；用分布式调度（K8s CronJob / Temporal Cron / serverless schedule）。

#### 决策矩阵 — 异步任务

| 场景 | 进程内 | Redis 队列 | 专用 Broker | 工作流引擎 | Cron |
|---|---|---|---|---|---|
| 临时小任务 | ✅ | ◯ | ❌ | ❌ | ❌ |
| 后台任务（中等量） | ❌ | ✅ | ◯ | ◯ | ◯ |
| 跨服务事件 | ❌ | ◯ | ✅ | ◯ | ❌ |
| 长周期业务流程 | ❌ | ⚠️ | ◯ | ✅ | ❌ |
| 周期任务 | ❌ | ◯ | ◯ | ◯ | ✅ |

## 12. 业务自动化与采集工程

> 本节只做架构层决策。RPA、OCR、视觉质检、数据治理、浏览器自动化和 POC 的具体切片与验证门禁见 `scenario-playbooks.md`。

### 12.1 官方 API / 数据导出

- **一句话定位**：通过平台提供的 API、导出任务或 webhook 获取数据。
- **适用场景**：平台有稳定 API；合规边界清晰；需要长期运行。
- **反例**：API 字段缺失且业务必须拿 UI-only 数据。
- **主要代价**：权限申请、配额限制、版本变更；有些平台商业 API 成本高。
- **退出成本**：低。协议清晰，迁移到其他来源相对容易。
- **常见组合**：API client + raw payload 存储 + normalized tables + rate limit。
- **决策信号**：有正式文档、可拿到 token、字段覆盖 P0 需求。
- **常见踩坑**：直接把 API response 当业务表用；应保留 raw，再做标准化。

### 12.2 浏览器自动化（Playwright / Selenium）

- **一句话定位**：模拟用户在 Web 页面上访问、点击、截图、导出。
- **适用场景**：没有 API；页面结构稳定；需要登录后数据；可接受维护成本。
- **反例**：强反爬、验证码频繁、平台条款不允许自动化。
- **主要代价**：selector 脆弱；浏览器环境依赖；账号/登录态管理；反爬和合规风险。
- **退出成本**：中。要迁到 API 或桌面 RPA 时，证据与 schema 可复用，操作脚本需重写。
- **常见组合**：授权商家后台用 `undetected-chromedriver` + Selenium + XPath selector registry + bounded random sleep；常规 Web 自动化用 Playwright + trace/screenshot + job table + raw HTML + normalized output。
- **决策信号**：目标流程可在固定 viewport 稳定跑；失败可观测；账号授权明确。
- **实现提示**：中国电商商家后台默认不要从 plain Chromium 起步；动作节奏要有随机 sleep、任务级限流和明确等待条件；元素定位优先 XPath + 文本/层级锚点，避免易变 class 和绝对第 N 个 div。
- **常见踩坑**：只存最终 CSV，不存 screenshot/HTML/trace，页面变更后无法定位；高频点击/滚动/导出导致账号状态异常。

### 12.3 桌面 RPA

- **一句话定位**：自动操作桌面客户端、企业软件或本地文件系统。
- **适用场景**：数据只在桌面客户端里；有控件树或稳定 UI；需要本地账号/证书。
- **反例**：纯 Web 能解决；客户端频繁改版；操作有高风险不可无人值守。
- **主要代价**：跨 OS 差异；窗口焦点、分辨率、权限；难以 CI。
- **退出成本**：中-高。应把业务动作抽象成 domain command，避免脚本散落。
- **常见组合**：Accessibility API / pywinauto / AppleScript + screenshot fallback + operator handoff。
- **常见踩坑**：把“点击第几个按钮”写死，没有前置状态判断和失败截图。

### 12.4 手机 RPA / 移动设备自动化

- **一句话定位**：通过真实或受控移动设备执行接近人工路径的 App 操作、截图、滚动和采集。
- **适用场景**：中国电商公域数据、App-only 数据、Web 端风控过强、直播间/短视频/移动端详情页等。
- **反例**：平台官方 API 或商家后台导出能稳定满足需求；没有账号授权或合规确认。
- **主要代价**：设备池、账号状态、风控、验证码、截图识别、失败重放；CI 和并发能力弱。
- **退出成本**：中-高。应把“采集意图”和“设备动作”拆开，避免业务逻辑绑定具体坐标。
- **常见组合**：Appium + ADB / iOS automation + OCR + screenshot/recording evidence + job table + operator handoff。
- **决策信号**：公域数据必须通过移动端可见路径获取；浏览器自动化失败率高；需要保留人工路径证据。
- **常见踩坑**：把手机 RPA 当作绕过合规的技术手段。它只能降低风控难度，不能替代授权、平台 ToS 评估和风险记录。

### 12.5 OCR / 截图识别采集

- **一句话定位**：无法稳定访问结构化数据时，从截图、PDF 或图片中恢复文本和结构。
- **适用场景**：报告、合同、账单、检测单、桌面客户端文本不可复制、直播评论截图等。
- **反例**：有稳定 API/DOM；关键字段要求 100% 自动正确且无法人审。
- **主要代价**：识别误差、版面差异、模型依赖；需要置信度和人工复核。
- **退出成本**：中。输出 schema 和样本集可复用，OCR 引擎可替换。
- **常见组合**：PDF/image render + OCR lines/blocks + schema extraction + confidence + review queue。
- **资源选型**：如果 OCR 只是后台基础能力且服务器只有 2C2G/2C4G，优先 Tesseract 或轻量 document loader OCR 集成，并通过异步 job 限流；PaddleOCR 适合 OCR 是核心能力、准确度要求高、中文复杂版面或必须版面恢复的项目。
- **常见踩坑**：LLM 直接读 OCR 文本产出结论，没有 bbox、置信度、字段校验。

### 12.6 抓包 / 协议逆向

- **一句话定位**：通过网络请求观察非公开接口并复用协议。
- **适用场景**：仅限有明确授权的内部系统、测试环境或客户允许的专有流程。
- **反例**：第三方平台条款禁止；涉及个人数据/敏感数据；无法确认授权。
- **主要代价**：合规风险高；协议变更不可控；证书/加密/风控复杂。
- **退出成本**：高。必须有合规记录和替代路径。
- **常见组合**：只在 design doc 中作为候选，默认要求 owner 明确批准。
- **常见踩坑**：把“能抓到”当“能上线”。上线前必须有授权、审计、限流和回滚。

#### 决策矩阵 — 数据获取方式

| 场景 | 官方 API | 浏览器自动化 | 桌面 RPA | 手机 RPA | OCR/截图 | 抓包/逆向 |
|---|---|---|---|---|---|---|
| 长期生产采集 | ✅ | ◯ | ◯ | ◯ | ⚠️ | ⚠️ |
| 一次性 POC | ✅ | ✅ | ◯ | ◯ | ◯ | ⚠️ |
| 合规敏感客户 | ✅ | ◯ | ◯ | ◯ | ◯ | ❌ |
| UI-only 数据 | ❌ | ✅ | ✅ | ✅ | ◯ | ⚠️ |
| 中国电商公域数据 | ⚠️ | ◯ | ❌ | ✅ | ◯ | ❌ |
| 商家管理后台 | ✅ | ✅ | ❌ | ◯ | ◯ | ⚠️ |
| 文档/图片数据 | ❌ | ❌ | ❌ | ❌ | ✅ | ❌ |

## 14. 前端状态管理

### 14.1 本地 state（useState / signals）

- **适用**：组件级；表单局部状态。
- **常见踩坑**：把跨页状态硬塞 props drilling。

### 14.2 全局 store（Redux / Zustand / Pinia / Jotai）

- **Redux Toolkit**：大型团队、强可追溯。
- **Zustand / Jotai**：轻量、最小 boilerplate。
- **Pinia**：Vue 生态默认。
- **决策**：状态范围 + 团队习惯。优先轻量。

### 14.3 服务端状态（TanStack Query / SWR / RTK Query）

- **一句话定位**：把"远程数据"作为一类特殊状态独立管理。
- **关键认知**：服务端状态 ≠ 客户端 UI 状态，必须分开管。
- **常见踩坑**：把服务端数据塞进 Redux 还自己写 loading/error/cache → 重新发明 TanStack Query。

### 14.4 表单（react-hook-form / formik / VeeValidate）

- 优先 react-hook-form（性能 + 心智）。
- 常见踩坑：每次输入都 re-render 整表单。

### 14.5 实时协作（CRDT / Yjs / Liveblocks / Automerge）

- **适用**：多人协作编辑、实时光标、共享白板。
- **代价**：CRDT 心智、网络层、冲突合并的边界。
- **常见踩坑**：把整文档作为单 CRDT，规模大后性能崩；分块。

| 远程数据为主 | ◯ | ⚠️ | ✅ | ◯ | ❌ |

## 15. 测试策略

### 15.1 测试金字塔 vs 蜂窝

- **金字塔**：大量单元 + 中量集成 + 少量 E2E。经典稳态。
- **蜂窝（trophy）**：以集成测试为主体。Web 全栈框架更适合。
- **决策**：业务复杂度高、单元重要 → 金字塔；React + 全栈 → 蜂窝。

### 15.2 单元 / 集成 / E2E / 视觉 / 契约

- **单元**：纯函数、领域逻辑；快、稳。
- **集成**：模块间真实交互；真 DB / fake 外部。
- **E2E**：Playwright/Cypress；少而关键。
- **视觉**：Chromatic / Percy；UI 库或设计敏感场景。
- **契约**：Pact / OpenAPI 校验；前后端独立部署必备。
- **常见踩坑**：E2E 写太多导致 CI 慢且翻牌；保留关键 happy path + 重灾区。

### 15.3 工具栈

- JS/TS：Vitest / Jest / Playwright。
- Python：pytest + httpx + factory_boy。
- Mock：MSW（前端）/ respx（Python）。

#### 决策矩阵 — 测试投资

|---|---|---|---|---|---|

## 16. CI / CD

### 16.1 平台

- GitHub Actions（默认起步）；GitLab CI（GitLab 用户）；CircleCI（高级缓存）；Buildkite（自托管 runner）。
- 决策：托管商内置 + 团队熟悉度。

### 16.2 分支策略

- **Trunk-based**：短期分支 + 频繁合并 main + feature flag。适合高成熟度团队。
- **GitHub flow**：main + 短期分支。最简单。
- **Git flow**：main/develop/release/hotfix。重发布周期、传统软件分发。
- 决策信号：发版频率、团队大小、回滚成本。

### 16.3 发布

- **直接发** → **Canary** → **Blue/Green** → **Feature Flag**。
- **Feature flag**：Unleash / LaunchDarkly / 自建。
- **常见踩坑**：feature flag 不清理 → 代码迷宫；设过期时间 + owner。

### 16.4 流水线分层

- PR 检查：lint + typecheck + 单元 + 受影响集成。
- Main：全量 + 部署到 staging + 冒烟。
- Release：tag + canary + 自动回滚阈值。

#### 决策矩阵 — CI/CD

| 场景 | Trunk-based | GitHub flow | Git flow | Canary | Feature flag |

## 17. 配置与密钥

### 17.1 .env / dotenv

- **适用**：单机/本地开发；少量 env。
- **反例**：生产密钥；多环境管理。

### 17.2 集中配置（Consul / etcd / 自家 config service）

- **适用**：多服务共享配置；动态配置（无重启变更）。
- **代价**：单点风险；客户端缓存一致性。

### 17.3 密钥管理（Vault / AWS Secrets Manager / Doppler / 1Password Secrets）

- **适用**：生产密钥、轮换、审计。
- **常见踩坑**：把密钥写进 docker image / git；用扫描工具。

### 17.4 环境分层

- 至少：dev / staging / prod；测试数据隔离；prod 不能从 dev 反向流。

#### 决策矩阵 — 配置/密钥

| 多服务生产 | ❌ | ◯ | ✅ |

## 20. 性能与扩展性预案

### 20.1 缓存层级

- 浏览器缓存 → CDN → 边缘 KV → 应用进程内（LRU）→ Redis → DB。
- 决策：每层都要明确 TTL 与失效路径。

### 20.2 限流与熔断

- 限流：令牌桶 / 漏桶；网关层 + 应用层双保险。
- 熔断：客户端断路器；下游故障不蔓延。
- 常见踩坑：限流策略只在网关，应用内调用未保护。

### 20.3 水平 vs 垂直扩展

- 水平：无状态服务首选。
- 垂直：DB 主写、Redis；尽量推迟分片。

### 20.4 异步化

- 把非实时路径改异步（邮件、报表、计算）。
- 常见踩坑：用户期望同步反馈，被改异步后没看到结果。

### 20.5 容量与压测

- 起步先压；每 quarter 复测。
- 常见踩坑：用真实生产配置之外的 staging 压测，结果不可信。

#### 决策矩阵 — 性能切入点

| 症状 | 第一动作 | 第二动作 |
|---|---|---|
| 列表慢 | 加索引 / 缓存 | 读写分离 |
| 下单慢 | 异步化非关键 | 队列削峰 |
| 全站慢 | CDN / 缓存 | 边缘函数 |
| 偶发 5xx | 限流 + 熔断 | 容量扩容 |
| 数据爆炸 | 归档 + 分区 | 分库分表 |
