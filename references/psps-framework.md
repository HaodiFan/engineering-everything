# PSPS Framework

PSPS 是 Engineering Everything 的通用构建洞察框架，用来把“用户只说了意图”转成可构建、可验证、可运营的系统。它不只适用于多维表，也适用于软件、组织、企业机制、SOP、知识库、自动化和非工程项目。

## 1. 定义

| 字段 | 含义 | 最小回答 |
|---|---|---|
| Persona | 谁在使用、决策、执行、验收 | 角色、权限、关注点 |
| Scenario | 在什么场景/流程/时间点发生 | 触发条件、频率、流转路径 |
| Pain | 当前卡在哪里、怕什么失败 | 痛点、风险、成本、信息缺口 |
| Solution Surface | 系统需要暴露什么能力或产物 | 页面/接口/流程/SOP/看板/自动化/指标 |

PSPS 的目标不是写更多字段，而是找出“必须构建什么才算闭环”。

## 2. 默认用户画像

面向这类用户时，agent 必须同时满足三种张力：

| 用户特征 | 意味着 | agent 应对 |
|---|---|---|
| 贪心的管理者 | 要数据完整、信息准确、能追责 | 流程要闭环，字段覆盖关键决策点和统计口径 |
| 不善言辞的需求者 | 说不清楚，但熟悉业务执行 | 从模糊输入里还原真实场景，主动补全 PSPS 初稿 |
| 懒惰的执行者 | 讨厌繁琐，要简单直接 | 必填项最少，默认值/自动化/模板优先，复杂字段给管理侧 |

设计规则：**管理字段可以完整，执行入口必须简单**。不要让一线执行者为管理者的贪心填一堆无用字段；能从流程、状态、日志、自动计算得到的字段，不让人手填。

## 3. PSPS 构建顺序

1. **先抽 Persona**：区分管理者、执行者、协作者、客户、系统/agent。
2. **再还原 Scenario**：入口、触发、频率、流转、异常、完成条件。
3. **再定位 Pain**：信息缺失、状态不透明、重复劳动、交接丢失、质量不可控、时间不可见、责任不清。
4. **最后定 Solution Surface**：需要数据模型、页面/视图、流程/SOP、自动化、指标、权限、通知、审计中的哪些。

不要先选技术栈、页面或工具；先用 PSPS 判断系统真实闭环。

## 4. 通用触发规则

| 命中信号 | 必须考虑的 Solution Surface | 通用工程解释 |
|---|---|---|
| Persona 包含管理者 / 老板 / owner | 统计/看板/报表 + GroupBy/过滤/指标口径 | 管理者要看进度、状态分布、核心指标和异常 |
| Scenario 涉及任务分配 / 审批 / 流转 / 交接 | 状态机 / Kanban / 队列 / owner / SLA | 任何流转都要状态、责任人、下一步和阻塞原因 |
| Pain 涉及图片 / 素材 / 商品 / 文件 / 资产 | 资产库 / Gallery / 元数据 / 搜索筛选 / 生命周期 | 资产管理首先是可发现、可预览、可追踪 |
| Scenario 涉及排期 / 计划 / 时间线 / 依赖 | Gantt / Calendar / Timeline / 起止日期 / 依赖 | 时间类系统必须能看计划、延期、依赖和关键路径 |
| Pain 涉及信息不准 / 口径不一 | 数据字典 / 字段 owner / 校验规则 / 审计日志 | 先统一口径，再谈统计 |
| Pain 涉及重复劳动 | 自动化 / 默认值 / 批量操作 / 模板 | 让系统替人做重复动作 |
| Pain 涉及质量不可控 | Checklist / Review gate / 抽检 / 验收标准 | 质量要有检查点，不靠口头提醒 |
| Pain 涉及协同混乱 | RACI / 通知 / 评论 / 变更记录 | 协同系统必须明确责任、沟通和历史 |
| Persona 包含外部客户 / 供应商 | 权限边界 / 可见范围 / 外部入口 / 脱敏 | 外部角色先考虑可见性和授权 |
| Scenario 涉及 agent/AI 自动处理 | 人审队列 / 置信度 / replay / fallback | AI 输出必须能被追踪、复核和兜底 |

## 5. 字段设计规则

### 必填字段

只保留执行闭环必需字段：

- 唯一标识 / 标题
- 当前状态
- owner / 责任人
- 触发来源
- 下一步动作
- 截止时间或完成条件（如涉及时间）
- 验收结果或失败原因

### 管理字段

用于统计和追责，但不一定让执行者手填：

- 优先级、业务线、客户/项目、成本、风险等级
- 状态变更时间、处理时长、延期原因
- 质量分、通过率、返工次数
- 指标口径、统计周期、数据来源

### 禁止字段

- “备注 1 / 备注 2 / 其他信息”这类无 owner 字段。
- 只为未来想象场景准备的字段。
- 需要执行者重复填写但可由系统推断的字段。

## 7. 完成判定

- Persona 至少覆盖“谁决策、谁执行、谁验收”。
- Scenario 至少覆盖主流程、异常流程和完成条件。
- Pain 能解释为什么需要这个系统，而不是功能堆砌。
- Solution Surface 能直接推导出数据字段、页面/流程、自动化和验证门禁。
- 对执行者的必填输入足够少；对管理者的统计口径足够完整。
