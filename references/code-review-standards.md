# Code Review Standards

用于代码审查、PR review、实现方法评审。目标不是只找编译错误，而是判断代码是否可长期维护、是否有潜在行为风险、是否能被测试和真人验收验证。

## 审查优先级

1. **代码优雅与坏味道**：参考 `refactoring-rules.md` 的坏味道规则库，优先指出会增加理解、修改、测试成本的问题。
2. **潜在问题与测试重点**：指出可能出问题的行为路径，并给出应该重点验证的自动化测试或真人验收场景。
3. **密钥、密码、硬编码风险**：检查是否有 secret 泄漏、示例配置误导、生产逻辑硬编码、日志落敏感信息。

## 输出要求

审查结果优先按发现排序，严重问题在前。每条问题尽量包含：

- 位置：文件与行号。
- 坏味道或风险类型。
- 为什么会影响维护或行为。
- 建议测试重点；最终用户流程以真人测试结果为真源。
- 如涉及密钥/硬编码，明确区分开发期 mock、示例配置、生产风险。

如果某个实现符合当前产品约定，应把它列为设计前提，不要误报。例如：权限由本地配置文件中的 Feishu open_id 列表判断时，审查重点应是配置解析、actor 传递、默认拒绝、测试覆盖，而不是要求改成数据库权限模型。

## 代码坏味道检查

本节只负责识别问题。命中坏味道并需要建议改法时，读取 `refactoring-rules.md`，按“坏味道 → 手法 → 前置测试 → 是否阻塞 PR”输出。

- **过长函数 / 过大类**：一个 service 同时处理 HTTP 编排、状态机、文件解析、外部同步、缓存、后台线程。
- **发散式变化**：一个需求变化会迫使同一个文件多处修改。
- **霰弹式修改**：一个小行为变更需要跨 router、service、schema、UI 多处同步，缺少单一策略层。
- **重复代码 / 重复规则**：前后端各自写一份限制、状态、角色、字段映射。
- **基本类型偏执**：大量裸字符串表示状态、角色、action、错误码，缺少 enum 或 transition table。
- **临时字段 / 神秘命名**：只在某些流程出现的 JSON 字段没有 owner 和 schema。
- **过度耦合**：业务 use case 直接知道外部 API 细节、文件系统细节、后台线程细节。
- **注释补丁味**：注释解释复杂绕路，而不是把逻辑抽成清楚的概念。
- **半成品混入主路径**：开发中功能直接接入生产入口，没有 feature flag、配置门禁或明确验收边界。

## 潜在问题与测试重点

按 feature 维度给测试建议：

- **权限/身份**：未登录、普通用户、管理员、CEO、未知 Feishu open_id、配置变更后重启。
- **状态机**：每个允许/禁止迁移都应有测试；并发点击、重复回调、重试、撤回、失败恢复要覆盖。
- **外部集成**：超时、非 2xx、部分成功、幂等重复、回调重放、签名/token 错误。
- **迁移**：空库升级、已有数据升级、downgrade 策略或不可 downgrade 说明。
- **前后端契约**：schema 字段缺失、旧数据兼容、枚举新增、前端 build/typecheck。
- **真人测试真源**：自动化测试证明回归风险；关键业务流最终以真人走通验收为准，记录账号、数据样本、步骤、结果。

## 密钥与硬编码检查

- 禁止真实 secret、token、密码、私钥进入 repo。
- `.env.example` 和 demo config 只能放空值或显式假值，不应放可用 secret。
- 本地开发 mock 必须默认关闭，或只在 dev/test 配置生效。
- 权限 hardcode 要区分：
  - **可接受**：产品明确用本地配置文件维护 CEO/admin Feishu open_id。
  - **不可接受**：生产逻辑写死具体个人 open_id、密码、token，或缺少配置说明。
- 日志不得记录 Authorization、cookie、token、signature、password、secret、完整回调敏感 payload。
